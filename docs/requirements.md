# 요구사항 정의서

## 1. 프로젝트 개요
- 정보 공유 및 파악
- 주요 사용자층 : 학생

## 2. 핵심 기능
- 신입생이나 아직 전공을 정하지 못한 학생들을 위한 정보 공유. 어떤 일을 하고 어떻게 하는 지 추천하는 시스템
- GSM 관련 교칙들을 데이터베이스에 저장 후 학생이 질문을 하면 교칙을 학습한 ai가 답을 해주는 시스템
- 매년 학사 일정을 갱신, 학생들이 질문하거나 일정이 다가오면 알려주는 시스템
- 메신저봇처럼 사용자가 입력을 하면 거기에 챗봇이 답하는 형식
- 예외 상황일 때 
    1. ai 응답 실패 : 응답에 실패 했으므로 wifi나 다른 것을 확인해달라고 또 다시 질문해달라고 요청
    2. 외부 api 장애 : api 문제로 불러오기를 실패했다고 wifi 등을 확인해달라고 하기
    3. 질문 내용이 부족하거나 부적절할 때 : 다시 입력을 요청
    4. 권한 없는 사용자의 요청 : 권한 확인 후 권한이 없다고 대답, 질문에는 답하지 않기
    5. 데이터가 없을 때 : 정보가 부족해 대답하지 못했다고 정중히 사과

## 3. 사용자 시나리오
- 학생이 서비스에 접속한다.
    회원 가입을 통해 기본 정보(학번, 전공 등 매년 자동 혹은 수동 갱신)를 얻는다.
    이를 기반으로 학생의 질문에 답한다. 특히 학사 가이드는 기본 정보를 더욱 이용한다.
    질문자는 챗봇의 답을 듣고 마음에 들지 않거나 더 궁금한 것이 있으면 추가 질문을 할 수 있다.

- 관리자가 로그인 한다.
    관리자 페이지에서 잘못도니 정보를 수정하거나 오류를 해결한다.
    학사 규정, 일정, 기업 정보나 실무 프로젝트 정보 등은 대부분 api를 이용해 자동 갱신한다.
    api 사용이 불가능 한 것들은 관리자가 직접 갱신한다.
    학생의 질문 데이터나 갱신된 정보들은 데이터 베이스에 필요한 정보만 저장된다.

## 4. 비기능 요구사항
- 최대 10초이내에는 대답을 권장함. 필요 시 시간을 더 사용 가능하나, 너무 오래 걸리는 시간 (20초 이상)은 답변에 실패했다고 한다.

- 인증은 gsm.hs.kr 이메일을 사용하는 사용자만 회원 가입이 가능하며, 이메일에 코드를 보내 이메일 진위 여부를 확인하다.

- 안정적으로 응답할 수 있어야 한다.
    AI 응답 요청이 몰릴 경우, 요청을 대기열 방식으로 처리하여 서버 과부하를 방지한다.
    서버 장애가 발생해도 핵심 기능(학사 정보 조회, 규정 설명)은 최소한의 형태로 지속 제공할 수 있어야 한다.
    데이터베이스는 단일 장애 지점이 없도록 백업 및 복구 절차를 마련한다.
    서비스는 향후 사용자 수 증가에 따라 서버를 수평 확장할 수 있는 구조로 설계한다.


## 5. 제약 조건
- api는 나중에 제공하는 api만 사용한다. 만약 api를 제공하지 않았다면 임시로 다른 신빙성있고 명성있는 곳의 api를 사용할 수 있지만 나중에 제공된 api와 바꿀 수 있어야 한다.
- 대부분의 정보를 수집한다. 회원가입을 할 때 데이터를 수집할 것이라고 명시할 것이다. 하지만 개인 정보는 최소한으로만 저장 (다른 사람에게 개인 정보 제공 금지)
    민감 정보는 마스킹
- 클라우드 : 클라우드 환경에 배포 가능해야 한다.
    지역 : 서울 (ap-northeast-2)
    도메인/HTTPS : 나중에
    API 키 관리 : .env
    보안 정책 : 민감한 정보는 안전하게 관리해야 한다.
    예산 : 원 $30~40
